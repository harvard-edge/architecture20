Appendix D — Architecture 2.0 Resource Links

Published

July 4, 2026

These links are not a general computer-architecture, pedagogy, or reproducibility directory. A resource earns space here only if it is directly useful for AI-mediated architecture work, such as naming a task, representing state, exposing actions, returning feedback, preserving evidence, or rejecting a result. Use the list as a starting point and check current versions before relying on any benchmark, dataset, simulator, or tool. This directory uses live links rather than formal bibliography entries; cite the primary paper or project artifact when making a scholarly claim.

D.1 Architecture 2.0 Framing

  • Essay: Architecture 2.0 gymnasium essay. Intent-level framing for why architecture needs data-centric environments rather than isolated model demonstrations.

  • Foundations article: IEEE Computer article. Background article for the Architecture 2.0 vision, autonomy levels, and participant roles in modern computer system design.

  • Workshop: ISCA 2026 Architecture 2.0 workshop. Community feedback loop for preview-edition claims, examples, benchmarks, and shared artifacts.

  • Portal: Architecture 2.0 resource site. Companion material and update surface for links that move faster than the book.

D.2 Architecture 2.0 Templates and Artifacts

  • Design-loop card and review rubric: sec-appendix-b-design-loop-card. Use the card to review a paper, proposal, research artifact, or internal loop before trusting its architectural claim.

  • Research paper card view: Table tbl-research-paper-card-view. Use the Architecture Claim Card view to state claim boundaries, workloads, baselines, evidence, negative traces, rejection authority, residual risk, and human decision ownership for one paper claim.

  • Worked Lighthouse claim-card view: Table tbl-lighthouse-claim-card-view. Use the example to see how the running XR prompt becomes an appendix-ready claim record without overstating illustrative evidence.

  • Blank design-loop template: Table tbl-blank-design-loop-card. Use the blank card as a one-page handout for paper reviews, project proposals, and design reviews.

  • Resource role checklist: sec-appendix-c-resource-catalog. Use the catalog to decide whether a benchmark, tool, dataset, or artifact actually makes the loop more representable, actionable, measurable, or auditable.

D.3 Roadmaps and Evidence Standards

  • IRDS Systems and Architectures: IRDS roadmap area. System-to-device roadmap from the International Roadmap for Devices and Systems (IRDS) that connects application drivers, system requirements, architecture trends, and technology constraints. Use it as a model for Architecture 2.0 roadmap plots that show capability gaps, evidence needs, and maturity horizons rather than only current tool lists.

  • IRDS Applications Benchmarking: IRDS editions. Application-driven benchmarking roadmap. Use it as a reference point when a loop claims that a workload, scenario, or benchmark suite is representative enough to support a broader architecture decision.

D.4 Architecture Reasoning and Design-Problem Benchmarks

  • QuArch: quarch.ai. Architecture question-answering and reasoning benchmark. Use it to test whether a model can reason over architecture concepts, but not as evidence that a loop can act through tools or reject candidates.

  • CVDP benchmark: NVlabs/cvdp_benchmark and Hugging Face dataset. Comprehensive Verilog design problems for RTL design and verification. Use it when the loop claim involves HDL generation, test harnesses, simulation feedback, or verification failures.

  • VerilogEval: NVlabs/verilog-eval. Specification-to-RTL and Verilog code-generation benchmark with executable checks. Use it for method-role claims about RTL generation and compile/test feedback, not for system-level architecture claims.

  • KernelBench: ScalingIntelligence/KernelBench. GPU-kernel generation benchmark with correctness and performance evaluation. Use it to study software-loop feedback, codegen, and performance evidence before claiming architecture-level benefit.

D.5 Architecture Environments and Design-Space Exploration

  • ArchGym: Architecture Gym. Environment interface for ML-assisted architecture design. Use it as a concrete example of actions, observations, costs, and feedback in a bounded architecture loop.

  • Timeloop: NVlabs/timeloop. Mapping and modeling tool for tensor workloads on accelerator architectures. Use it for dataflow, mapping, memory hierarchy, and accelerator design-space loops.

  • Accelergy: Accelergy. Energy-estimation infrastructure for accelerators. Use it when a loop needs an explicit energy feedback source and calibration boundary.

  • MAESTRO: maestro-project/maestro. Analytical cost model for DNN dataflows and tiling. Use it as a fast-feedback model whose limits must be recorded before higher commitment.

D.6 Full-System Simulation and Hardware/Software Harnesses

  • gem5: gem5 simulator. Modular computer-system simulator. Use it for architecture feedback that needs workload execution, timing behavior, and reproducible simulator state.

  • FireSim: FireSim. FPGA-accelerated full-system simulation. Use it when the loop needs stronger hardware/software feedback than a software-only proxy can provide.

  • Chipyard: Chipyard docs. Integrated framework for generating and evaluating hardware systems. Use it when the loop must connect generators, RTL, simulation, and implementation artifacts.

D.7 Physical-Design and EDA Evidence

  • OpenROAD: OpenROAD project. Open-source RTL-to-GDS flow. Use it for loops that need physical-design feedback, timing/area/power evidence, or signoff-adjacent rejection.

  • CircuitNet: CircuitNet. VLSI CAD dataset for machine-learning applications in EDA. Use it for cross-stage prediction and physical-design learning claims, while preserving tool provenance and task scope.

  • ChiPBench: AI chip-placement benchmark. Benchmark focused on end-to-end physical-design impact for AI chip placement. Use it when placement evidence must be tied to downstream physical metrics, not only intermediate scores.

D.8 Workload and Benchmark Governance

  • XRBench: XRBench paper. Extended-reality machine-learning benchmark suite. Use it as a workload anchor for mobile-XR architecture loops, including scenario definition and workload coverage questions.

  • MLCommons benchmarks: MLCommons benchmarks. Benchmark governance and reporting infrastructure. Use it as a model for workload versions, run rules, comparability, and community-maintained evidence rather than as a generic performance leaderboard.

D.9 AI Evidence and Reproducibility Practice

  • ML reproducibility checklists: NeurIPS reproducibility program. Checklist culture for making assumptions, experiments, compute, and evidence explicit. Use it as an adjacent-field template when writing Architecture 2.0 evidence ledgers.

  • Hugging Face Papers: Hugging Face Papers. Successor surface to the Papers with Code convention of linking papers, tasks, datasets, and code (the original service was retired in 2025). Use it as a reminder that architecture needs comparable task and artifact metadata, not only isolated results.

  • Software Heritage: Software Heritage archive. Long-term source-code archival infrastructure. Use it when provenance matters more than a mutable repository link.

D.10 Case Studies to Build

  • Learned chip placement: Pair sec-feedback-verification-trust with sec-loop-patterns-across-stack to read placement claims as evidence ledgers: task, baseline, tool flow, physical metrics, rejection authority, and independent reproducibility.

  • Architecture gymnasiums: Pair sec-architecture-environments-tool-interfaces with sec-methods-generation-prediction-optimization to read ArchGym-style work as an environment contract. It covers actions, observations, costs, and what can reject a result.

  • Kernel and code-generation loops: Pair sec-loop-patterns-across-stack with KernelBench-style resources to ask when a software-loop result becomes architecture evidence and when it remains only local code evidence.

D.11 Using and Citing the Preview

Use this preview as a working vocabulary for paper reviews, project proposals, class discussions, and internal design reviews. For citation, cite the preview URL and version when referring to this book directly, and cite the Architecture 2.0 foundations article when referring to the broader vision and autonomy framing. For feedback on the preview, use the ISCA 2026 Architecture 2.0 workshop, the Architecture 2.0 resource site, or the arch2 repository issues.
